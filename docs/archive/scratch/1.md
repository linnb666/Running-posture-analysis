一、总体原则（必须遵守）

分析引擎（running/modules/）保持独立：Web 通过 import / pip-editable 调用，不混入 HTTP 代码。

以“可复现 / 可追溯 / 可比较”为核心目标，UI 的价值是“把可复现性用视觉证据呈现出来”。

开发优先级：MVP 优先、创新分阶段。不要一开始上 Celery / 分布式，先做轻量可运行方案。

不要盲目切换框架：如果你熟 Flask，MVP 用 Flask；在文档里保留 FastAPI 迁移路径（可选 P2）。

七、关键实现细节 / 工程建议（避免踩坑）

输入文件 hash：保存 video sha256，避免重复计算，作为追溯键。

配置快照：将用于本次分析的 config_json（所有阈值、model, fps assumptions）保存到 DB。

模型版本化：模型文件名 + sha256 + human readable version（如 motionbert-v1.2-rm_ap3d_sha256）并展示在 UI。

不可变输出目录：每次分析保存到 outputs/{task_id}，包括 report.html, metrics.json, frames/, video_with_skeleton.mp4。

小批量并发规划：用线程池 + semaphore 控制最大并发任务（例如 1 GPU）。

错误可观测性：Worker 在遇到异常时写入 analysis_records.status='failed' 和 traceback.txt，并推送到前端。

单元/集成测试：写 smoke tests：上传小测试视频，等待完成，检查 metrics_json 存在。
