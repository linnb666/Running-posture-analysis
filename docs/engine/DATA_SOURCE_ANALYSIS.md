# 跑步姿态分析系统 - 数据来源与计算方法详解

## 文档目的
本文档详细说明系统中每个分析指标的数据来源（2D/3D）、计算方法和已知问题，
便于问题排查和系统优化。

---

## 数据流概览

```
视频输入
    ↓
MediaPipe Pose (2D检测)
    ↓
├─→ 2D关键点序列 (33个MediaPipe关键点)
│       ↓
│   视角检测 (view_detector.py)
│       ↓
│   ├─→ 侧面视角分析
│   └─→ 正面视角分析
│
└─→ MotionBERT (2D→3D提升)
        ↓
    3D关键点序列 (17个H36M格式关键点)
        ↓
    运动学分析 (kinematic_analyzer.py)
```

---

## 一、侧面视角 (Side View) 指标

### 1. 步频 (Cadence)
| 属性 | 值 |
|------|-----|
| **数据来源** | **2D** (MediaPipe) |
| **关键点使用** | 左右脚踝Y坐标 (landmarks[27], landmarks[28]) |
| **计算方法** | 检测脚踝Y坐标峰值（触地时刻），计算峰值数/时长×60 |
| **代码位置** | `kinematic_analyzer.py:3055-3082` (`_cadence_from_ankle_position`) |
| **可靠性** | 高 (侧面视角脚踝Y变化明显) |

**计算公式**:
```
step_count = len(left_peaks) + len(right_peaks)
cadence = (step_count / duration) * 60  # spm
```

**潜在问题**:
- 快速跑者步频高时，峰值检测distance参数可能过滤部分有效峰值
- prominence阈值0.01可能对小振幅动作不敏感

---

### 2. 膝关节角度 (Knee Angle)
| 属性 | 值 |
|------|-----|
| **数据来源** | **仅3D** (MotionBERT) |
| **关键点使用** | H36M格式: l_hip, l_knee, l_ankle / r_hip, r_knee, r_ankle |
| **计算方法** | 3D向量夹角，投影到矢状面(XY平面) |
| **代码位置** | `kinematic_analyzer.py:317-346` (`_calculate_angles_by_phase`) |
| **可靠性** | 依赖3D质量 |

**重要说明**:
- **2D不计算膝角** - 代码明确标注："膝关节角度只在3D数据可用时计算"
- 如果3D数据不可用，系统返回空数据并显示警告

**计算公式**:
```python
# 3D膝关节角度（投影到XY平面后计算）
v1 = hip - knee
v2 = ankle - knee
cos_angle = dot(v1, v2) / (|v1| * |v2|)
knee_angle = arccos(cos_angle)
```

**已知问题**:
- AP3D微调后膝角RMSE仍有61°误差（评估结果）
- 3D姿态在腿部交叉/遮挡时可能不准确

---

### 3. 落地膝角 (Landing Knee Angle)
| 属性 | 值 |
|------|-----|
| **数据来源** | **3D** (膝角) + **2D** (落地时刻检测) |
| **检测方法** | 2D脚踝Y峰值确定落地区间，3D膝角取峰值前的帧 |
| **生物力学约束** | 140° ≤ angle ≤ 170°, 伸展趋势(rate≥-1) |
| **代码位置** | `kinematic_analyzer.py:844-1019` |
| **可靠性** | 中等 (依赖落地检测+3D膝角双重准确) |

**潜在问题**:
- 落地区间检测基于2D脚踝Y坐标，侧面视角较可靠
- 如果3D膝角本身不准，落地膝角也会不准

---

### 4. 垂直振幅 (Vertical Amplitude)
| 属性 | 值 |
|------|-----|
| **数据来源** | **侧面2D主导**，3D仅用于姿态约束 |
| **关键点使用** | 2D: hip中心Y为主，肩中心Y辅助；3D仅用于姿态一致性参考 |
| **计算方法** | 去趋势后按步态周期统计身体中心Y振荡幅值，并归一化为躯干长度百分比 |
| **代码位置** | `kinematic_analyzer.py:1126-1274` |
| **可靠性** | 侧面中高（受拍摄质量和镜头抖动影响） |

**重要改进**:
- 侧面视角默认使用**2D主信号**计算垂直位移，避免root-relative 3D压低振幅
- 归一化分母采用稳健版2D躯干参考长度
- 通过去趋势与周期分段，降低镜头抖动和局部噪声影响

**评分标准**:
```
优秀: ≤11% 躯干长度
良好: 11-17%
一般: 17-25%
待改进: >25%
```

---

### 5. 触地时间 (Ground Contact Time)
| 属性 | 值 |
|------|-----|
| **数据来源** | **2D** (MediaPipe) |
| **关键点使用** | 左右脚踝Y坐标 (landmarks[27], landmarks[28]) |
| **计算方法** | 脚踝Y峰值周围的"触地区间"持续时间 |
| **代码位置** | `kinematic_analyzer.py:1439-1601` |
| **可靠性** | 侧面视角高，正面视角**不可用** |

**触地区间检测逻辑**:
```python
# 阈值 = 峰值 - 振幅 * 30%
# 从峰值向前后搜索，找到低于阈值的边界
# 触地时长 = (end_frame - start_frame) * frame_duration_ms
```

**评分标准**:
```
精英: <210ms
优秀: 210-240ms
良好: 240-270ms
一般: 270-300ms
较差: >300ms
```

---

### 6. 前倾角度 (Forward Lean / Body Lean)
| 属性 | 值 |
|------|-----|
| **数据来源** | **3D优先 + 2D一致性约束**，不足时回退2D |
| **关键点使用** | 双肩中点到双髋中点的连线 |
| **计算方法** | 支撑相关帧上的躯干中线与垂直线夹角，采用截尾均值稳健统计 |
| **代码位置** | `kinematic_analyzer.py` |
| **可靠性** | 侧面中高（受拍摄质量、机位偏差和骨盆轴稳定性影响） |

**计算公式**:
```python
# 侧面前倾角（支撑相关帧）
lean_angle = angle(trunk_midline, vertical)
# 先计算单帧3D前倾，再用2D一致性约束，最后对支撑相关帧做截尾均值
```

**评分标准**:
```
最优: 4-8°
良好: 3-10°
一般: 2-12°
过直立: <2°
过度前倾: >12°
```

---

### 7. 手臂摆动 (Arm Swing)
| 属性 | 值 |
|------|-----|
| **数据来源** | **2D** (MediaPipe) |
| **关键点使用** | 肩-肘-腕三点角度 (landmarks[11,13,15] / [12,14,16]) |
| **计算方法** | 肘关节角度的活动范围(max - min) |
| **代码位置** | `kinematic_analyzer.py:3441-3482` |
| **可靠性** | **低** (特别是远离镜头侧) |

**已知严重问题**:
```
⚠️ 侧面视角下，远离镜头的手臂可能：
1. 被身体遮挡，MediaPipe visibility低
2. 检测到但坐标不准确
3. 完全丢失检测

→ 这可能是用户反馈"远侧手臂摆动几乎没有"的原因
```

**计算公式**:
```python
# 2D肘关节角度
v1 = shoulder - elbow
v2 = wrist - elbow
arm_angle = arccos(dot(v1,v2) / (|v1| * |v2|))
amplitude = max(angles) - min(angles)
```

**潜在改进方向**:
1. 检查远侧关键点visibility，若低于阈值则不输出/标记为不可靠
2. 使用3D数据计算手臂摆动（当前未实现）

## 二、正面视角 (Front View) 指标

### 1. 步频 (Cadence - Frontal)
| 属性 | 值 |
|------|-----|
| **数据来源** | **2D** (MediaPipe) |
| **关键点使用** | 左右膝X坐标差值 (landmarks[25], landmarks[26]) |
| **计算方法** | FFT主频提取 + 自相关验证 |
| **代码位置** | `kinematic_analyzer.py:1866-1958` |
| **可靠性** | 中-高 (FFT方法物理意义明确) |

**算法原理**:
```python
# 1. 提取左右膝X坐标差值信号
x_diff = left_knee_x - right_knee_x

# 2. 带通滤波 (1.2-4.0 Hz，对应72-240 spm范围)
filtered_signal = bandpass_filter(x_diff, 1.2, 4.0)

# 3. FFT提取主频
peak_freq = fft_peak_frequency(filtered_signal)
cadence = peak_freq * 60 * 2  # 每个周期左右各一步

# 4. 自相关验证
cadence_autocorr = autocorrelation_peak_period(filtered_signal)
```

### 2. 下肢力线 (Lower Limb Alignment)
| 属性 | 值 |
|------|-----|
| **数据来源** | **2D** (MediaPipe) |
| **关键点使用** | 髋-膝-踝 (landmarks[23,25,27] / [24,26,28]) |
| **计算方法** | 膝关节相对于髋-踝连线的横向偏移 |
| **代码位置** | `kinematic_analyzer.py:2518-2672` |
| **可靠性** | 高 (正面视角核心指标) |

**计算公式**:
```python
# 计算膝关节相对于髋-踝连线的横向偏移
hip_ankle_x = (hip['x'] + ankle['x']) / 2
knee_offset = knee['x'] - hip_ankle_x
# 转换为角度
valgus_angle = arctan(knee_offset / hip_ankle_dist)

# 正值 = 膝外翻 (valgus)
# 负值 = 膝内扣 (varus)
```

**评估标准**:
```
正常: -2° ~ +2°
轻度偏移: ±2° ~ ±5°
中度偏移: ±5° ~ ±8°
严重偏移: >±8°
```

## 三、3D骨架生成相关问题

### 3D骨架视频生成流程
```
2D MediaPipe关键点
    ↓
MediaPipe→H36M映射 (17个关键点)
    ↓
MotionBERT DSTformer (2D→3D提升)
    ↓
3D骨架坐标 (T, 17, 3)
    ↓
骨架绘制和视频渲染
```

### 已知3D问题

#### 问题1: 腿部摆动不准确
**可能原因**:
1. AP3D微调后膝角RMSE仍有61°
2. MotionBERT对快速动作/遮挡情况处理不佳
3. 2D检测时腿部交叉造成的跟踪混乱

**数据证据** (评估结果):
```
Per-Joint MPJPE改进:
- right_knee: 74.19% (0.1053→0.0272)
- left_knee:  80.12% (0.1506→0.0299)
- right_ankle: 82.56% (0.2215→0.0386)
- left_ankle: 72.90% (0.1610→0.0436)

虽然改进明显，但绝对误差仍然存在
```

#### 问题2: 远侧手臂摆动缺失
**可能原因**:
1. MediaPipe在侧面视角对被遮挡的手臂检测不稳定
2. 2D检测缺失导致3D推断无效
3. MotionBERT对单侧输入的处理

**排查建议**:
1. 检查远侧关键点的visibility值（应该<0.5）
2. 检查3D输出中对应手臂的坐标是否合理
3. 对比原始视频和3D骨架的手臂位置

## 五、问题排查检查清单

### 用户报告: "远侧手臂摆动几乎没有"
1. [ ] 检查视频中远侧手臂是否被身体遮挡
2. [ ] 检查MediaPipe输出中远侧手臂关键点visibility (<0.5为低)
3. [ ] 检查3D骨架中对应手臂的X/Z坐标是否异常
4. [ ] 对比左右手臂的`arm_swing.left_arm_range`和`right_arm_range`

### 用户报告: "腿部摆动比较差劲"
1. [ ] 检查AP3D评估中的膝角RMSE (当前61°)
2. [ ] 检查视频中是否有腿部交叉/遮挡
3. [ ] 检查2D检测中腿部关键点的visibility
4. [ ] 检查3D输出中腿部关键点的时序连续性

### 数据准确性问题
1. [ ] 确认视角检测是否正确 (`view_angle`)
2. [ ] 确认3D数据是否可用 (`has_3d_data`)
3. [ ] 检查`data_reliability`字段的标注
4. [ ] 对比2D和3D计算的结果差异
